Match rules architecture

JCMF2026 · 4 June 2026

module seam leakage deep module

Reviewing whether the current architecture supports adding qualifying species filters and alternative win conditions to competitive matches.

Key finding: The win condition is implicit — hardcoded as total-weight sorting across 4+ files. Neither qualifying species nor win-condition strategy exists as a concept in the type system or data flow. Adding them would touch every layer from config UI to leaderboard rank, demanding a MatchRules seam.

Problem

Win condition is implicit — total weight baked into every layer. Adding "biggest fish wins" or "points by classification" requires editing 6+ files across the stack. Species filtering doesn't exist at all.

Solution

Introduce a MatchRules module with a WinCondition strategy and SpeciesFilter. PrepState stores the rules. GameState passes fish events through the rules. Leaderboard queries the rules for rankings. One seam, two adapters per strategy.

Wins

  • Leverage — one interface (WinCondition), N implementors (total weight, biggest fish, points, fish count)
  • Locality — scoring logic in one module, ranking in one module, UI config in one component
  • Test surface — MatchRules tested in isolation without GameState instantiation
  • Deletion test passes — deleting total-weight logic today scatters the same accumulation across 4 call sites

Problem

Four sites duplicate the same "record caught fish" logic — creating a CaughtFish entry, pushing to arrays, accumulating weight. Adding a new scoring field (e.g. points, species-specific count) means editing all four.

Solution

Extract a Scorecard module behind the MatchRules seam. A single scorecard.recordCatch(angler, fish) call replaces the four manual blocks. Works with any win condition.

Wins

  • Locality — one module owns the "how a fish becomes points" question
  • Leverage — one interface, 4 call sites replaced by 1
  • Test surface — scorecard tested in isolation; loop and state tests stop re-testing accumulation
  • Enables — species-specific scoring (points by classification, qualifying species filters) naturally lands here

Problem

Leaderboard sorts by totalWeightOz at line 42, hardcoded. The multiplayer path duplicates the sort at line 32 in +page.svelte. Adding a new win condition means editing both sorts and the derived data structures.

Solution

Extract a Ranker module that accepts MatchRules and returns pre-ranked entries. Leaderboard becomes a pure display component — no sorting logic, no derived.

Wins

  • Locality — ranking logic testable without Puppeteer
  • Leverage — one Ranker used by both bot-match and multiplayer paths
  • Test surface — no more "is the sort direction right?" inside a Svelte component test

Top recommendation

Candidate 1 — Introduce a MatchRules module. This is the prerequisite for candidates 2 and 3. Without it, every new win condition or species filter must be hand-stitched across 6+ files. With it, adding "biggest fish wins" or "carp-only" becomes writing a second adapter behind an existing seam. Start here: define the WinCondition and SpeciesFilter interfaces, implement the total-weight adapter (move existing logic behind it), then add the UI picker in prep/rules. Each subsequent condition is a new file, not a cross-cutting edit.

Candidate 2 (Scorecard) is a natural second step — it consolidates the 4 duplicate accumulation blocks into one call. Do it after the MatchRules seam exists so the Scorecard sits behind it.
